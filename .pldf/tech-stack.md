# Технологический стек: Визуализатор кратчайших маршрутов

**Дата создания**: 21.05.2026  
**Тип проекта**: веб (SPA, клиентское приложение)

## Выбранные технологии

### Язык программирования
**Выбор**: JavaScript (ES2022+), среда выполнения — браузер  
**Обоснование**: UI, граф на SVG, алгоритм Дейкстры и анимация работают в одном месте; не нужен сервер и обмен данными по сети. Проще сдать курсовую на оценку «3».

### Frontend
**Фреймворк/Библиотека**: без React/Vue — **Vanilla JavaScript** + **Vite 6** (сборка и dev-сервер)  
**Обоснование**: прототип уже на HTML/SVG/CSS; фреймворк не даёт выгоды для одного экрана с холстом. Vite даёт модули, hot reload и простой `npm run build`.

**Дополнительные библиотеки**:
- Нет обязательных UI-библиотек — стили по утверждённому прототипу (чистый CSS)
- **SVG DOM API** — отрисовка графа (встроено в браузер)

### Backend
**Не используется** — расчёт и состояние графа полностью на клиенте (согласно концепции P1).

### База данных
**Не используется** — граф и маршрут хранятся в памяти приложения; сохранение в файл исключено из scope.

### Инструменты тестирования
**Unit-тесты**: **Vitest** — алгоритм Дейкстры, построение пути по сегментам, структура графа (ориентированный/неориентированный)  
**Integration-тесты**: не планируются на первом этапе  
**E2E-тесты**: не планируются (ручная проверка UI в браузере)

### Инструменты сборки и развертывания
**Сборка**: Vite (`npm run build` → статика в `dist/`)  
**Развертывание**: любой статический хостинг или локально `npm run preview` / `dist/` через `python -m http.server`

### Дополнительные инструменты
- **Node.js 20+** / **npm** — установка зависимостей и скрипты Vite/Vitest
- **ESLint** (опционально) — единый стиль кода
- **Git** — версионирование (коммиты выполняет студент)

## Подход к организации кода

**Выбранный подход**: **Layer-first** (по слоям)  
**Обоснование**: чёткое разделение доменной логики (граф, алгоритм) и UI; проще тестировать `algorithms/` и `graph/` без браузерного DOM; соответствует учебной курсовой.

**Планируемая структура `src/`**:
```
src/
  graph/        # модель графа, вершины, рёбра
  algorithms/   # Дейкстра, сборка маршрута по сегментам
  state/        # состояние приложения, режимы инструментов
  ui/           # SVG-холст, панели, модалки, анимация
  main.js       # точка входа
```

## Ограничения и предпочтения

- Только **JavaScript** в браузере (без Python-бэкенда) — решение студента после обсуждения.
- **Vite** для разработки и сборки.
- **Vitest** для тестов логики.
- Без React/Vue, без БД, без сохранения файлов (P3 исключён).
- Веса рёбер ≥ 0 (ограничение алгоритма Дейкстры).

## Альтернативы, которые рассматривались

| Компонент | Альтернатива | Почему не выбрали |
|-----------|--------------|-------------------|
| Стек | JavaScript + Python (Flask) | Лишний API и деплой; UI всё равно на JS |
| Сборка | Файлы без Vite | Хуже масштаб модулей и тестов |
| UI | React / Vue | Избыточно для одного экрана |
| Тесты | Без автотестов | Студент выбрал Vitest для надёжности алгоритма |
| Структура | Feature-first | Студент выбрал слои для разделения логики и UI |
| Отрисовка | Canvas API | В прототипе SVG; проще подписи весов и клики по рёбрам |

## Ключевые решения этапа

### 1. Клиентский JavaScript без backend

- **Причина**: весь сценарий (редактор, Дейкстра, анимация) выполняется в браузере; Python не упрощает задачу.
- **Влияние**: architecture — монолитное frontend-приложение; deploy — только статика; plan — нет этапов API.

### 2. Vite + Vanilla JS

- **Причина**: модульная структура и dev-сервер без сложности React; соответствует HTML-прототипу.
- **Влияние**: architecture — `src/` с ES-модулями; implement — настройка `vite.config` и `index.html` в корне.

### 3. Layer-first + Vitest

- **Причина**: тестируемая логика в `graph/` и `algorithms/`; UI отдельно в `ui/`.
- **Влияние**: architecture — явные границы слоёв; plan — отдельный шаг «тесты Дейкстры» до связывания с SVG.

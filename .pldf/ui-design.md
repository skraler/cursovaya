# Дизайн пользовательского интерфейса: Визуализатор кратчайших маршрутов

**Основано на**: [.pldf/concept.md](concept.md)

## Обзор

Одностраничное веб-приложение в **светлом схематичном** стиле: белый холст с сеткой, тёмно-синие рёбра, зелёная подсветка маршрута. Три колонки: инструменты слева, холст по центру, маршрут и управление справа. Все функции P1 и P2 сосредоточены на главном экране; P3 (сохранение, экспорт) **исключены** по решению заказчика.

## Прототип интерфейса

**Главный файл**: [ui-prototype/index.html](ui-prototype/index.html)

**Рабочий экран**: [ui-prototype/workspace.html](ui-prototype/workspace.html)

**Запуск**: см. [ui-prototype/README.md](ui-prototype/README.md) — `python -m http.server 8000` в папке `ui-prototype`.

## Основные экраны

### Главный экран: Редактор графа и маршрут

**Файл**: [ui-prototype/workspace.html](ui-prototype/workspace.html)  
**Приоритет**: P1 + P2

**Назначение**: построение графа, задание маршрута, расчёт Дейкстры по сегментам, визуализация и анимация результата.

**Элементы интерфейса**:

| Зона | Элементы |
|------|----------|
| Левая панель (P1) | Вершина, Ребро (2 клика), Старт, Остановка, Финиш |
| Левая панель (P2) | Переместить, Удалить вершину/ребро, Очистить холст |
| Центр | SVG-холст с сеткой, вершины, рёбра, веса, стрелки для ориентированных рёбер |
| Правая панель | Список маршрута, суммарная длина, сегменты, «Построить маршрут», «▶ Пуск», «Сбросить подсветку» |
| Модальное окно | Вес ребра, тип (ориентированное / неориентированное) |

**Данные на экране**:
- Отображаемые: граф, метки вершин, веса рёбер, очередь маршрута, длина по сегментам и итого, статус анимации
- Вводимые: координаты вершин (клик), пара вершин для ребра, вес и тип ребра в модалке, клики для старт/остановки/финиш

**Действия пользователя**:
- Добавить вершину кликом на холст
- Создать ребро двумя кликами → модалка веса и направления
- Назначить старт, промежуточные точки, финиш кликом по вершинам (порядок в списке справа)
- «Построить маршрут» → подсветка кратчайшего пути зелёным
- «▶ Пуск» → пошаговая плавная анимация закрашивания рёбер маршрута
- P2: редактирование и очистка через инструменты слева

## Пользовательские потоки

### Поток 1: Построить маршрут с остановками

**Цель**: получить кратчайший путь A → B → C → D на нарисованном графе.

1. Пользователь добавляет вершины и рёбра на холсте.
2. В режимах «Старт», «Остановка», «Финиш» кликает по вершинам A, B, C, D.
3. Справа отображается упорядоченный список.
4. Нажимает «Построить маршрут» — рёбра сегментов подсвечиваются, показывается длина 14.
5. Опционально: «▶ Пуск» — рёбра по очереди плавно зеленеют.

**Альтернатива**: если сегмент недостижим → красный баннер с текстом ошибки.

### Поток 2: Анимация для демонстрации (курсовая)

1. После построения маршрута пользователь нажимает «▶ Пуск».
2. Каждые ~700 мс подсвечивается следующее ребро маршрута (плавный transition CSS).
3. По завершении все рёбра маршрута остаются зелёными.

### Поток 3: Редактирование графа (P2)

1. Пользователь выбирает «Переместить» / «Удалить» / «Очистить».
2. Выполняет действие на холсте (в реализации — с подтверждением для очистки).

## Данные и состояния

### Данные
- **Вершина**: id, x, y, роль (none | start | waypoint | finish)
- **Ребро**: from, to, weight, directed
- **Маршрут**: упорядоченный массив id вершин
- **Результат**: список рёбер полного пути, длины сегментов, total

### Состояния UI
- **Режим инструмента** (vertex | edge | start | …) — подсказка на холсте
- **Маршрут построен** — рёбра класса `route`
- **Анимация** — поочерёдно класс `animated` на рёбрах
- **Ошибка недостижимости** — видимый `error-banner`

## Обработка ошибок

- Нет пути между точками сегмента → баннер в правой панели, расчёт не завершается.
- Отрицательный вес → валидация в модалке (min=0), сообщение при реализации.
- Пустой маршрут (нет старта/финиша) → кнопка «Построить» неактивна (в реализации).

## Адаптивность

Веб, desktop-first (курсовая демонстрация на ноутбуке). На узких экранах (<900px) панели перестраиваются в вертикальный стек.

## Ключевые решения этапа

### 1. Одностраничный макет «холст + две панели»

- **Причина**: граф и маршрут должны быть видны одновременно; переключение экранов мешает наглядности.
- **Влияние**: tech — SPA или один HTML+JS; architecture — единый state store для графа и UI-режима.

### 2. Светлый схематичный стиль и зелёная анимация рёбер

- **Причина**: ассоциация с чертежом/планом; зелёный цвет однозначно показывает «пройденный» участок при «Пуск».
- **Влияние**: tech — SVG + CSS transitions; реализация анимации как последовательность рёбер, не вершин.

### 3. Исключение P3 (сохранение / экспорт)

- **Причина**: явное требование заказчика упростить объём до оценки «3».
- **Влияние**: нет File API / backend для файлов; фокус плана реализации на P1+P2.
